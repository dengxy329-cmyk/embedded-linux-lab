# war-stories / 调试实战记录

> 每个工具至少破解过一个真 bug = M4 交付物。这里就是放那些"我破解过的 bug"。

## 命名规则

```text
NNN-<subsys>-<现象关键词>.md

001-i2c-rk3x-rcu-stall.md
002-gpio-rockchip-probe-defer-loop.md
003-spi-rockchip-dma-misalign.md
```

## 写作模板（重点是"假设链"，不是只写结论）

```markdown
# NNN-<subsys>-<现象关键词>

## 现象
- dmesg / 测试输出 / 客户描述
- 复现步骤（步骤越短越好）

## 工具
- 用了什么调试工具
- 哪个最关键

## 假设链 ⭐⭐⭐（这部分必写，是教训沉淀）
1. 假设 A：xxx → 验证方法：yyy → 结果：否决/命中
2. 假设 B：xxx → 验证方法：yyy → 结果：否决/命中
3. ...

## 根因
（一行话说清）

## 修复
- 改动 diff 或 commit hash
- 为什么这样修

## 教训
- 下次怎么早点定位
- 哪个工具下次先用
- 哪个假设以后可以更快否决
```

## 评分（自己给自己打）

| 维度 | 分数 | 说明 |
|------|------|------|
| 复现是否稳定 | / 5 | 5 = 100% 复现，1 = 偶发 |
| 工具用得对吗 | / 5 | 用对了首选工具拿满分 |
| 假设链长不长（短=好） | / 5 | 命中前的假设数 ≤ 3 拿满分 |
| 根因解释清不清 | / 5 | 一句话能讲清拿满分 |
| 教训能不能复用 | / 5 | 抽象成下次先看的 check list |
| **派别标签** | - | **实战派**（线上/项目真 bug）/ **教程派**（自己造的） |

> 教程派也算（M2/M3 必造一些 bug 练手），但写笔记时必须**标清楚**。
> 面试时讲 war story，对方第一句就问"线上的还是实验的"。

## 数量目标

- M5 结束：≥ 8 篇
- M6 结束：≥ 15 篇
